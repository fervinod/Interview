USE STAR METHOD:
What is the STAR method?
The STAR method is an interview technique that gives you a straightforward format you can use to tell a story by laying out the situation, task, action, and result.

Situation: Set the scene and give the necessary details of your example.
Task: Describe what your responsibility was in that situation.
Action: Explain exactly what steps you took to address it.
Result: Share what outcomes your actions achieved.



A STAR method example answer
It’s making sense now, isn’t it? Here’s one more example for some added clarity.

If the interviewer says: “Tell me about a time when you had to be very strategic in order to meet all of your top priorities.”

Your response might be (just, you know, don’t actually say “situation” etc):

Situation: “In my previous sales role, I was put in charge of the transition to an entirely new customer relationship management (CRM) system—on top of handling my daily sales calls and responsibilities.”
Task: “The goal was to have the migration to the new CRM database completed by the start of Q3, without letting any of my own sales numbers slip below my targets.”
Action: “In order to do that, I had to be very careful about how I managed my time. So I blocked off an hour on my calendar each day to dedicate solely to the CRM migration. During that time, I worked on transferring the data, as well as cleaning out old contacts and fixing outdated information. Doing this gave me enough time to chip away at that project while still handling my normal tasks.”
Result: “As a result, the transfer was completed two weeks ahead of deadline and I finished the quarter 10% ahead of my sales goal. The new CRM has also helped us get more organized as a team, and overall our department sales are up 25% year over year.”

If a production server is “crashed” and you **don’t have terminal access**, the goal is to quickly determine:

1. Is the machine actually dead?
2. Is it a network problem?
3. Is the OS hung?
4. Is hardware failing?
5. What’s the blast radius?
6. Can service be restored safely?

This is very similar to how a Meta Production Engineer or hyperscaler infra engineer would approach an incident.

# 1. Define the impact first

Before touching the machine:

Ask:

* Is one server affected or many?
* Is customer traffic impacted?
* Is redundancy still healthy?
* Is this isolated to one rack/AZ/cluster?

Check:

* monitoring dashboards
* alerts
* load balancer health
* service metrics
* latency/error spikes

Important:
A “dead server” may actually be:

* drained intentionally
* isolated by automation
* failing health checks
* unreachable only from one network path

---

# 2. Verify basic network reachability

Without shell access, networking is your first signal.

Check:

* ping
* traceroute
* DNS resolution
* TCP port connectivity
* load balancer health

Examples:

```bash id="g3z4lv"
ping server01
traceroute server01
nc -zv server01 22
curl http://server01:port/health
```

Interpretation:

* ping fails + traceroute dies early → network/routing issue
* ping works but SSH fails → service/OS/firewall issue
* TCP connects but app unhealthy → application problem

---

# 3. Check out-of-band management immediately

In real data centers, this is critical.

Use:

* IPMI
* iDRAC
* iLO
* BMC
* serial console
* KVM-over-IP

These often remain accessible even when the OS is dead.

Look for:

* power state
* POST failures
* kernel panic screen
* boot loops
* BIOS errors
* thermal alarms
* RAID/controller alerts

This is one of the highest-value diagnostic steps.

---

# 4. Determine whether the machine is:

* powered off
* kernel panicked
* hung
* reboot looping
* isolated by networking

Key signs:

| Symptom                | Likely Cause                   |
| ---------------------- | ------------------------------ |
| BMC unreachable        | power/network/rack issue       |
| BMC reachable, OS dead | kernel panic/hang              |
| Repeated reboots       | hardware/kernel failure        |
| SSH timeout only       | firewall/network/service issue |
| Console frozen         | kernel deadlock                |
| PXE/boot errors        | disk/storage issue             |

---

# 5. Check monitoring history before the crash

Very important.

Look at:

* CPU trends
* memory pressure
* disk latency
* network drops
* thermal sensors
* hardware alerts

You want:

> “What changed before failure?”

Common patterns:

* memory exhaustion
* kernel OOM
* disk filling
* NIC flapping
* RAID degradation
* thermal shutdown

---

# 6. Compare against healthy servers

Hyperscaler troubleshooting relies heavily on comparison.

Check:

* same rack
* same cluster
* same deployment batch
* same firmware version
* same kernel version

Questions:

* Did a rollout happen?
* Is this isolated to one hardware generation?
* Is a network switch involved?
* Shared dependency?

This often finds root cause faster than staring at one host.

---

# 7. Inspect orchestration / automation systems

Modern infra is heavily automated.

Check:

* deployment systems
* config management
* orchestration tools
* recent pushes
* health automation

Possible causes:

* bad kernel rollout
* broken config
* failed firmware update
* automation quarantined host

A surprising number of “crashes” are deployment-related.

---

# 8. Attempt console access

If available:

* serial console
* hypervisor console
* KVM session

Look for:

* panic traces
* filesystem corruption
* boot failures
* login prompts
* stuck services

Common signs:

* kernel panic
* emergency mode
* fsck failures
* init/systemd failures

---

# 9. Preserve evidence before rebooting

This is important in strong interviews.

Weak candidates:

> “I’d reboot immediately.”

Strong candidates:

* gather diagnostics first
* preserve logs if possible
* identify recurrence risk
* understand impact

Before reboot:

* capture screenshots
* save console logs
* collect BMC events
* note timestamps
* correlate alerts

---

# 10. Decide whether reboot is safe

Rebooting is mitigation, not diagnosis.

Questions:

* Is this stateful?
* Is failover healthy?
* Could reboot worsen corruption?
* Is root cause understood?
* Is hardware suspect?

Sometimes:

* reboot restores service temporarily
* but destroys debugging evidence

---

# 11. Escalate intelligently

Good production engineers communicate clearly.

Example:

> “Single host affected in rack R12. BMC reachable. Host appears stuck during kernel boot after recent firmware rollout. No customer impact due to redundancy.”

This matters a lot in Meta-style interviews.

They care less about commands and more about:

* structured reasoning

* operational judgment

# A very strong answer structure

For interviews:

> “First I’d determine impact and scope. Then I’d verify network reachability and check out-of-band management access. I’d inspect monitoring leading up to the failure, compare against healthy systems, and determine whether this is hardware, networking, OS, or deployment-related. Before rebooting, I’d preserve diagnostic information and evaluate service impact.”

That sounds much stronger than:

> “I’d try rebooting it.”
